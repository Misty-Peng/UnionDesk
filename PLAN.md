## 3. 后续开发阶段

### Phase 1: 业务域与权限收口

目标：把当前 demo 登录和静态权限，收敛成可维护的认证、授权与业务域范围控制。

重点任务：

- 补齐 `iam` 模块，明确用户、角色、业务域、授权关系。
- 将 `UserContext` 接入服务层，后端计算有效业务域范围。
- 对工单、咨询、附件等对象级权限做服务层复核。
- 管理端写操作补充审计入口，为后续留痕统一打底。
- 统一错误码、异常返回和请求追踪链路。

验收标准：

- 未登录请求稳定返回 401。
- 越权访问被后端拒绝，而不是前端隐藏按钮。
- 关键写接口具备清晰的权限边界。

### Phase 2: 工单主链路分层重写

目标：把现有 demo 工单流程改造成真正分层的核心业务链路。

重点任务：

- 拆分工单实体、状态流转、动态字段、优先级、SLA。
- 从控制器直连 demo 数据，改为 controller / service / repository 分层。
- 补齐列表、详情、创建、流转、备注、回复、事件日志。
- 让客户提交、客服处理、关闭工单形成完整闭环。

验收标准：

- 至少一条完整工单链路可本地回归。
- 工单状态变更可追踪。
- 关键操作有业务事件记录。

### Phase 3: 咨询、反馈、通知、审计

目标：补齐工单之外的核心支线能力，并与主链路打通。

重点任务：

- 咨询会话与转工单能力落地。
- 反馈/建议的提交与流转落地。
- 站内通知先完成，再预留邮件/短信扩展点。
- 审计日志具备查询与导出入口。

验收标准：

- 咨询、工单、审计三者之间可以串起追踪链路。
- 至少一种通知渠道可用。
- 核心管理动作能在审计中查到。

### Phase 4: 前端业务化与共享契约

目标：把两个前端从 demo 页面升级为真正的业务界面。

重点任务：

- 固化共享 API client 与类型契约。
- 收敛前端路由、登录态、错误处理、加载态。
- 客户端侧聚焦提交、查询、咨询、反馈。
- 管理端侧聚焦工单池、处理、配置、权限、审计。

验收标准：

- 前端不再重复定义请求类型。
- 两端共享同一套接口契约。
- 至少一条业务流程可以从前端走完到后端再返回前端。

### Phase 5: 测试、发布与可观测性

目标：让项目从“能跑”进入“可持续交付”。

重点任务：

- 后端补单测、切片测试和关键集成测试。
- 前端补关键流程测试。
- 构建、类型检查、测试纳入统一验证流程。
- 补日志、指标、健康检查和部署收口。

验收标准：

- 后端构建稳定通过。
- 前端构建稳定通过。
- 核心链路具备最少一轮本地回归。

## 4. 当前推荐执行顺序

1. 先做 Phase 1，把权限和业务域边界固定住。
2. 再做 Phase 2，把工单主链路分层重写。
3. 然后做 Phase 3，补齐咨询、反馈、通知、审计。
4. 接着做 Phase 4，收口前端契约与业务页面。
5. 最后做 Phase 5，把测试和发布能力补齐。

## 5. 当前风险

- 目前仍保留 demo 数据和 demo 接口，后续重构时要避免与正式分层长期并存。
- 前后端已经能联调，但接口契约还没有完全 OpenAPI 化。
- 业务域隔离和对象级权限还没有完全落地，后续如果拖延，重构成本会继续上升。
