# UnionDesk 开发计划

> 本计划基于当前仓库真实状态制定：`PRD`、`系统架构设计`、`数据库设计`、`技术栈方案` 已完成，代码工程仍处于启动阶段。计划遵循 `AGENTS.md` 的约束：先读后写、最小变更、目标驱动、逐步验证。

---

## 1. 计划目标

在不偏离现有设计文档的前提下，逐步把 UnionDesk 从“文档已齐备”推进到“可运行、可验证、可交付”的状态。

### 总目标
1. 建立后端与前端工程骨架。
2. 打通本地开发依赖与启动链路。
3. 完成鉴权与业务域权限的最小闭环。
4. 完成工单提交、处理、关闭的最小业务闭环。
5. 为后续 SLA、咨询转工单、通知能力预留扩展点。

## 3. 分阶段计划

### 阶段 0：工程骨架与开发环境

**目标**：让项目具备可启动、可配置、可验证的基础结构。

#### 任务 0.1 后端工程初始化
- 创建 `UnionDesk/` Spring Boot 工程。
- 接入基础配置、统一异常、统一返回结构、基础包结构。
- 接入 Flyway，并确保可执行初始化迁移。

**检查点**
- 后端工程可启动。
- Flyway 可从空库执行。
- 基础健康检查接口可访问。

#### 任务 0.2 前端 Monorepo 初始化
- 创建 `UnionDeskWeb/` workspace。
- 拆分 `apps/customer-web`、`apps/admin-web`、`packages/shared`。
- 建立基础路由、请求封装、类型共享、构建脚本。

**检查点**
- `pnpm install` 可执行。
- 两个应用能分别启动。
- `shared` 可被两个应用复用。

#### 任务 0.3 本地依赖环境
- 提供 `docker compose` 开发依赖。
- 覆盖 MySQL、Redis、MinIO。
- 提供最小的环境变量说明与启动路径。

**检查点**
- 依赖服务可一键启动。
- 后端可连接数据库与缓存。
- 文件上传链路具备基础存储能力。

---

### 阶段 1：身份认证与业务域权限

**目标**：让系统先具备“能登录、能识别域、能限制可见范围”的能力。

#### 任务 1.1 登录与 Token 刷新
- 实现 Access Token + Refresh Token。
- 建立登录态校验与刷新机制。
- 接入基础密码策略。

**检查点**
- 用户可登录。
- Token 过期后可刷新。
- 未登录请求被正确拒绝。

#### 任务 1.2 IAM 与业务域范围
- 实现用户、角色、域授权的基础模型。
- 后端按 `business_domain_id` 进行数据隔离。
- 不依赖前端传参作为权限判断依据。

**检查点**
- 不同业务域数据互不可见。
- 超级管理员可跨域。
- 域内用户仅可访问授权范围。

#### 任务 1.3 基础管理接口
- 业务域基础 CRUD。
- 最小用户与角色查询接口。

**检查点**
- 域配置可读可写。
- 管理端可看到当前授权范围。

---

### 阶段 2：工单最小闭环

**目标**：完成客户提单、客服处理、工单关闭的核心流程。

#### 任务 2.1 工单类型与动态字段
- 支持工单类型配置。
- 支持基础动态字段定义与校验。
- 保持运行态字段值可扩展。

**检查点**
- 域内可配置工单类型。
- 提单时字段校验有效。
- 无效输入能被拦截。

#### 任务 2.2 客户端提单与查单
- 客户端完成选域、填写动态表单、上传附件、提交工单。
- 支持查看本人提交的工单列表与详情。

**检查点**
- 客户可以成功提单。
- 客户可查看自己的工单。
- 附件上传流程可用。

#### 任务 2.3 管理端工单池处理
- 管理端展示工单池。
- 支持领取、分配、回复、关闭。
- 记录关键状态变化。

**检查点**
- 客服可以处理工单。
- 状态流转符合规则。
- 处理动作可审计。

#### 任务 2.4 审计日志
- 记录核心业务写操作。
- 记录管理端配置变更。

**检查点**
- 关键动作可追溯。
- 审计数据与业务数据可关联。

---

### 阶段 3：增强能力

**目标**：补齐产品可用性与运维能力。

#### 任务 3.1 SLA 与超时预警
- 配置首响时限、解决时限。
- 支持暂停计时状态。

**检查点**
- SLA 配置生效。
- 超时工单可被识别并告警。

#### 任务 3.2 咨询转工单
- 咨询会话可转为工单。
- 保留上下文与附件追溯。

**检查点**
- 会话与工单建立关联。
- 追溯链路完整。

#### 任务 3.3 通知模板与发送
- 先实现站内通知。
- 后续扩展邮件、短信。

**检查点**
- 关键事件可通知。
- 模板变量可渲染。

#### 任务 3.4 OpenAPI 驱动前端类型
- 由后端接口定义生成前端共享类型。
- 前端请求与响应统一使用共享层类型。

**检查点**
- 前端不手写重复接口类型。
- 接口变更可以统一同步。

## 4. 推荐执行顺序

如果当前要立即开始开发，建议顺序如下：

1. 初始化后端工程骨架。
2. 初始化前端 Monorepo 骨架。
3. 配置本地依赖服务。
4. 打通登录与权限。
5. 打通工单最小闭环。
6. 再补 SLA、通知、咨询转工单等增强能力。

这个顺序的好处是：先把平台能力搭起来，再逐步填业务，不会过早陷入复杂流程。

## 5. 验收标准

### 阶段 0 验收
- 后端、前端、依赖服务都能本地启动。
- 数据库初始化可从空库完成。

### 阶段 1 验收
- 用户登录可用。
- Token 刷新可用。
- 业务域权限隔离有效。

### 阶段 2 验收
- 客户提单可用。
- 客服处理可用。
- 工单关闭与审计可用。

### 阶段 3 验收
- SLA 可配置并生效。
- 咨询转工单可用。
- 通知链路可用。

---

## 6. 开发原则

- 只做当前阶段必须做的事。
- 先满足主流程，再补边界与增强。
- 每一步都要有可验证结果。
- 代码、文档、数据库设计保持一致。
- 若发现文档与实现冲突，优先停止并确认，而不是擅自改动。

---

1. 初始化 `UnionDesk/` 后端工程。
2. 初始化 `UnionDeskWeb/` 前端工作区。
