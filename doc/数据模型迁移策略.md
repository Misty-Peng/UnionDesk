# 数据模型迁移策略

> 目标：将现有 Flyway V1 基线（基于 `user_account` 单表模型）迁移至新 PRD V1.0 定义的 `identity_subject` + `staff_account` + `customer_account` 三表身份模型。

## 1. 新旧模型差异对照

### 1.1 身份与账号

| 旧表 (Flyway V1 基线) | 新实体 (PRD V1.0) | 变更说明 |
|--------------------|-------------------|----------|
| `user_account` | `identity_subject` + `staff_account` + `customer_account` | 拆分为统一身份主体 + 两张账号扩展表 |
| `user_account.username` | `staff_account.login_name` / `customer_account.login_name` | 分别存储 |
| `user_account.mobile` | `identity_subject.phone` | 手机号上提至身份主体，作为主匹配标识 |
| `user_account.employment_status` | 移除 | 员工状态通过 `staff_account.status` + `domain_member.status` 管理 |
| — | `identity_subject.merged_into_id` | 新增，身份归并指针 |
| — | `customer_cancel_request` | 新增，客户注销申请与恢复 |
| — | `identity_binding` | 新增，第三方身份绑定（云之家等） |

### 1.2 角色与权限

| 旧表 | 新实体 | 变更说明 |
|------|--------|----------|
| `role` (scope=global/domain 合一) | `platform_role` + `domain_role` | 拆分为平台角色和域角色两套；`platform_admin` 迁入平台角色，`super_admin` 迁入业务域角色 |
| `user_global_role` | `staff_account_platform_role` | 关联对象由 user_id 改为 staff_account_id |
| `user_domain_role` | `domain_member_role` | 关联对象改为 domain_member.id |
| `iam_permission` | `permission_item` | 简化，去除 http_method/path_pattern |
| `iam_role_permission` | `domain_role_permission` | 仅域内角色使用 |
| `iam_role_binding` | 移除 | 角色绑定拆入上述两张关联表 |
| `iam_admin_menu` | 前端路由 + 权限项控制 | 菜单不再存数据库，由前端权限组件驱动 |
| `iam_admin_role_menu_relation` | 移除 | 同上 |
| `platform_organization` | 移除 | PRD 不含组织树 |

| 旧表 | 新实体 | 变更说明 |
|------|--------|----------|

| `customer_business_domain_access` | `domain_customer` | 重命名，新增 source/activated_at 等字段 |
| — | `domain_member` | 新增，员工与域的关系 |
| — | `domain_config` | 新增，域级配置键值 |
| — | `consultation_config` | 新增，咨询专用配置 |
| — | `domain_template` | 新增，业务域模板 |
| — | `invitation_code` | 新增，邀请码 |
| — | `domain_customer_tag` | 新增，客户标签 |

### 1.5 咨询

## 2. 迁移策略

### 2.1 迁移原则
- 使用 Flyway 管理所有 schema 变更，**仅允许前向迁移**。
- 旧版 DDL 作为 Flyway V1 基线，后续变更通过增量迁移脚本完成。
- 迁移前必须完成数据库全量备份。

### 2.2 迁移步骤

#### 阶段 1：新建身份三表 + 数据迁移
1. 创建 `identity_subject`、`staff_account`、`customer_account` 三表。
2. 从 `user_account` 按 role 拆分数据到对应表。
3. 建立 `identity_subject.phone` ↔ `user_account.mobile` 映射。
4. 保留 `user_account` 旧表作为过渡引用（添加 `migrated_to_subject_id` 字段标记）。

#### 阶段 2：角色与权限迁移
1. 创建 `platform_role`、`domain_role`、`permission_item` 等新表。
2. 从 `role` + `iam_permission` 迁移数据。
3. 清理 `iam_role_binding`、`iam_admin_menu` 等旧表。

#### 阶段 3：业务表适配
1. 所有引用 `user_account.id` 的外键改指向对应新表。
2. `ticket.customer_id` → `customer_account.id`。
3. `ticket.assigned_to` → `staff_account.id`。
4. 咨询、反馈等表同理。

#### 阶段 4：清理
1. 确认无代码引用旧表后，删除 `user_account`、`iam_role_binding`、`iam_admin_menu`、`iam_admin_role_menu_relation`、`platform_organization` 等旧表。
2. 确认 Flyway 迁移脚本可从零构建最终态 schema。

### 2.3 风险与注意事项
- 迁移脚本必须**幂等**，可重复执行不报错。
- 外键约束需按依赖顺序处理，先建新表后迁移数据最后删旧表。
- 迁移过程中需保持应用兼容，建议分版本灰度：先双写，再切读，最后删旧。
- `user_account` 中存在 `email` 唯一约束允许 NULL，新模型中 `staff_account.email` 和 `customer_account.email` 同样允许 NULL。

## 3. 操作规范
- 每个迁移脚本文件命名：`V{版本号}__{描述}.sql`
- 迁移脚本需与应用版本一并纳入 CI/CD 流水线
- 发布前需在测试环境完整执行迁移并验证
- 生产执行前必须完成备份校验
