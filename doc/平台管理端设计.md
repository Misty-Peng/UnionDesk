# 平台管理端设计

> 当前状态：已接入平台入口、平台模式路由、平台首页真实统计与 P0 入口导航、平台用户真实列表、平台组织骨架和离职池真实列表；已增加 P0 契约对齐页：业务域管理（`/platform/domains`）、客户入域（`/platform/domain-onboarding`）、工单池、站内信、附件，以及可复用的 step-up 二次验证弹窗。平台菜单与平台角色当前复用系统侧成熟页面能力。导出中心、公告、日志、屏蔽词、知识库尚未完成，各模块内导入能力尚未开始。
> 当前平台与业务的分层原则是“同壳不同域”：进入平台模式后，菜单源、页签源和搜索结果都只跟随平台域，不再和业务域共用同一份状态。

## 1. 目标

平台管理端复用现有 `UnionDeskAdminWeb`，不再拆分新的前端应用，而是在同一个管理端里增加一个“平台模式”。

平台模式的目标是把平台级治理能力和业务域能力分层：

- 平台级：平台入口、组织治理、角色与菜单治理、导出中心、公告、日志、内容治理
- 业务域级：工单类型、动态字段、通知模板、业务域可见范围
- 系统级：登录、鉴权、审计、配置、可观测性

## 2. 当前实现

### 2.1 入口与切换

- 平台入口放在管理端头部右侧动作区，位置在头像左侧。
- 当前入口可见性按 `platform.*` 权限快照控制，未登录或无平台权限时不渲染。
- 点击入口后进入 `/platform/home`。
- 在平台模式下，同一位置切换为“返回业务端”。
- 平台态返回业务端采用同源整页切换，避免平台页签状态与业务页签状态在同一次路由跳转里互相干扰。
- 返回业务端默认回到 `VITE_BASE_HOME_PATH`，未配置时回到 `/system/menu`。

### 2.2 已接入页面

- `/platform/home`：平台管理面板骨架页，用于承接平台模式入口和后续模块导航。
- `/platform/user`：平台用户骨架页，当前展示平台用户列表结构、状态口径和批量治理动作占位。
- `/platform/dept`：平台组织页，当前接入平台内部组织树只读接口，展示真实组织树、负责人和组织节点字段；新增、编辑和模块内导入动作仍为占位，导出后续统一走平台导出中心。
- `/platform/import-export`：当前仍以历史路由承载导出中心骨架页，页面用于统一展示导出任务、结果下载与异步任务占位；真实任务流和审计链路待补齐。导入能力不再收口到该页，后续下沉到各模块内部。
- `/platform/domains`：业务域管理页，对齐 P0 `admin/domains` 字段与删除前 step-up；后端未发布管理域接口时列表回退只读演示域。
- `/platform/domain-onboarding`：客户入域页，邀请码与域客户表格对齐 P0 路径；后端未发布时表格为空。
- `/platform/ticket-pool`：管理端工单池，对齐 `admin/domains/{id}/tickets` 与领取接口；未发布时回退演示工单列表。
- `/platform/inbox`：站内信列表，对齐 `/inbox`；未发布时为空列表。
- `/platform/attachments`：附件联调入口，对齐 `attachments/presign` 与 `attachments/upload`。
- `/platform/offboard-pool`：离职池骨架页，当前展示离职人员列表、恢复/导出动作占位和离职生命周期字段。
- `/platform/menu`：平台菜单页，当前复用系统菜单管理页面能力，并在平台菜单域下打开独立页签。
- `/platform/role`：平台角色页，当前复用系统角色管理页面能力，并在平台菜单域下打开独立页签。
- `/system/menu`：业务侧菜单管理页，主内容背景已切换到 `bg-colorBgLayout`，跟随主题使用布局浅灰底。

### 2.3 当前权限模型

- 前端入口可见性当前已切换为 `platform.*` 权限快照。
- 后端和数据库已经具备 `platform.*` 权限元数据、统一角色绑定和 Admin 菜单树能力。
- 平台入口不再仅依赖管理员角色判断，和权限快照结果保持一致。

### 2.4 当前边界

- 已完成：入口、平台首页、平台模式切换、平台用户骨架、平台组织骨架、离职池骨架、菜单/角色复用、菜单页主题背景修正、平台/业务页签隔离。
- 进行中：平台组织真实接口。
- 未开始：导出中心真实任务链路、各模块内导入、公告、日志、屏蔽词、知识库。

## 3. 模块规划

### 3.1 平台治理

- 平台首页
- 用户管理
- 组织/部门管理
- 离职池
- 导出中心

平台治理模块负责承接平台级的人、组织和统一导出任务能力；各业务对象的导入能力在对应模块页内承接。

### 3.2 权限治理

- 角色管理
- 菜单管理
- 权限码目录
- 统一角色绑定

这一层与现有 IAM 和 Admin 菜单体系共用同一套权限模型。

### 3.3 业务配置

- 平台域管理
- 工单类型管理
- 动态字段配置
- 通知模板
- 公告中心

这一层主要负责业务域可配置项，当前已在数据库和后端预留，前端页面逐步补齐。

### 3.4 数据流转边界

- 导出能力集中在平台导出中心，统一承接异步任务、下载、失败结果和审计记录。
- 导入能力下沉到各模块页面内部，例如平台用户导入、平台组织导入、平台角色导入、平台域导入。
- 各模块页面应提供模板下载、导入入口、最近导入记录或结果反馈，不再额外建设统一导入大厅。

### 3.5 内容治理

- 屏蔽词管理
- 知识库管理

这两块当前还没有成品页面，后续需要单独补骨架和后端接口。

### 3.6 运维审计

- 日志中心
- 操作审计
- 系统配置
- 安全策略

当前项目里 `operation_log`、登录日志和会话表已经存在，前端审计页仍待落地。

## 4. 与数据库和后端的对齐点

- `user_account` 已规划登录名和离职字段，支持在职、停用、离职三种生命周期。
- `auth_login_config` 已接入滑块验证码开关和登录方式开关。
- `iam_permission`、`iam_role_permission`、`iam_role_binding` 已支持平台/域权限分层。
- `iam_admin_menu` 继续承载 Admin 菜单树，后续平台页直接复用这套导航和授权能力。
- 当前前端路由命名仍以单数形式为主，数据库和历史迁移中存在部分旧的复数路由种子，后续需要统一。

## 5. 开发方向

1. 先把 `/platform/dept` 从静态骨架数据接入真实组织接口，再补组织新增、编辑和模块内导入链路。
2. 再补导出中心和平台域管理。
3. 然后补公告、日志、屏蔽词、知识库。
4. 最后统一前端单数路由、历史复数种子和 Flyway 迁移脚本的口径，再继续补平台域管理与其余治理模块。

## 6. 与实现同步的约定

- 已完成的事项要在 checklist 上打钩。
- 只写“已落地”和“明确预留”的内容，不把骨架页写成成品页。
- 与代码和数据库迁移存在差异时，优先在文档里明确标注，不要默认读者能猜到。
