## 1. 产品定位与范围

### 1.1 产品定位
UnionDesk 是一个面向企业客户服务场景的多业务域工单平台，统一处理咨询、工单、反馈/建议三类请求。

### 1.2 仓库边界（按你的规划）
- `UnionDesk`：后端服务项目，负责 API、鉴权授权、业务规则、通知和审计。
- `UnionDeskWeb`：前端项目，包含两个应用：
  - 客户端：客户提单、查单、反馈。
  - 管理端：客服处理、配置管理、权限管理。

### 1.3 MVP 目标
1. 实现多业务域的数据与权限隔离。
2. 支持每个业务域独立配置工单类型与动态字段。
3. 打通客户提交到客服处理的闭环流程。
4. 提供 SLA 规则与超时预警。
5. 预留知识库二期扩展接口。

### 1.4 MVP 不包含
- 智能客服自动回复正式上线。
- 完整 BI 报表体系。
- 外部供应商协作门户。

## 2. 角色与职责

| 角色 | 核心职责 |
|---|---|
| 客户 | 提交咨询/工单/反馈，查看本人进度 |
| 客服（Agent） | 领取、处理、回复、转派、关闭工单 |
| 业务域管理员（Domain Admin） | 管理本域工单配置、SLA、通知模板、域内权限 |
| 超级管理员（Super Admin） | 管理业务域、全局策略、跨域授权 |

---

## 3. 核心流程

### 3.1 提交工单
1. 客户选择业务域与工单类型。
2. 填写动态表单并上传附件。
3. 后端校验字段规则并创建工单。
4. 工单进入队列并通知相关客服。

### 3.2 处理工单
1. 客服领取或被分配工单。
2. 进行回复、内部备注、状态流转、优先级调整。
3. 若信息不足，流转至“待客户补充”。
4. 解决后关闭并留存审计记录。

### 3.3 咨询转工单
咨询支持一键转工单，保留会话上下文与附件追溯。

### 3.4 反馈/建议
反馈与工单解耦，使用独立状态流（待评估/处理中/已采纳/已拒绝），但复用业务域权限。

## 4. 功能需求

### 4.1 业务域管理
- 基础属性：`code`、名称、LOGO、主题色、状态。
- 可见策略：全局可见、白名单可见、申请后可见。
- 支持统一入口与独立域入口。

### 4.2 工单核心能力
- 预置类型 + 自定义类型。
- 动态字段支持文本、单选、多选、数字、日期时间、附件、级联。
- 字段属性支持必填、客户可见、默认值、排序、校验规则。
- 状态流转可配置，并强制校验合法流转。
- 状态变化、指派变化、关键操作必须审计。

### 4.3 SLA 与通知
- SLA 按业务域 + 工单类型配置（首响时限、解决时限）。
- 支持暂停计时状态（例如待客户补充）。
- 默认站内通知，可按域启用邮件/短信。
- 通知模板支持变量（如 `ticket_id`、`domain_name`）。

### 4.4 权限模型
- 采用 RBAC + 业务域范围控制。
- 后端强制 `business_domain_id` 过滤，禁止仅依赖前端参数。
- 超级管理员可跨域，客服仅可见授权域。

### 4.5 知识库预留（二期）
- 预留工单关闭后生成知识草稿接口。
- 预留知识条目审核流（待审/通过/拒绝）。

---

## 5. 非功能需求

### 5.1 安全
- JWT + Refresh Token。
- 密码使用 Argon2 或 bcrypt。
- 管理操作必须记录审计日志（人、时、对象、结果）。

### 5.2 性能
- 工单列表常用查询 P95 < 1 秒。
- 创建/流转接口 P95 < 500ms。

### 5.3 可靠性
- 核心能力可用性目标 99.9%。
- 每日备份并定期恢复演练。

### 5.4 可观测性
- 监控指标：时延、错误率、SLA 达成率、通知成功率。
- 全链路透传 `request_id`。

---

## 6. 数据设计约束

1. 使用“全局账号 + 域内授权”模型。
2. 避免依赖 MySQL 复合唯一索引中 `NULL` 语义保证唯一性。
3. 动态字段采用“配置表 + JSON 值 + 热字段索引”的混合方案。
4. 核心实体统一审计列：`created_at`、`updated_at`、`created_by`、`updated_by`。

---

## 7. MVP 验收标准

- 至少 3 个业务域可独立运行并通过权限隔离验证。
- 每个业务域可独立配置至少 5 类动态字段。
- 客户端可完成“提交 -> 补充 -> 关闭”全流程。
- 管理端支持分配、转派、状态流转、SLA 预警。
- 关键操作可追溯到审计日志。
- API 文档、错误码文档、部署文档齐备。
