# 系统架构设计文档（UnionDesk）

| 版本 | 日期 | 负责人 | 说明 |
|---|---|---|---|
| V1.3 | 2026-04-06 | 架构组 | 明确前端采用 React + Ant Design，日志采用 ELK 优先与本地文件回退 |

---

## 1. 架构目标与原则

### 1.1 架构目标
- 支撑多业务域客户服务场景的数据和权限隔离。
- 支撑 `UnionDesk`（后端）与 `UnionDeskWeb`（客户端+管理端）协同研发。
- 在 MVP 阶段兼顾交付速度与可维护性。

### 1.2 核心原则
1. 先模块化单体，再按边界拆分服务。
2. 默认按业务域进行访问控制。
3. 配置驱动业务行为（工单字段、SLA、通知模板）。
4. 可观测性内建，不后补。

---

## 2. 仓库与应用边界

### 2.1 后端仓库：`UnionDesk`
职责：
- 身份认证与会话管理。
- IAM（用户、角色、域授权）与权限中间件。
- 工单、咨询、反馈、通知、审计核心能力。
- 对前端提供统一 REST API。

### 2.2 前端仓库：`UnionDeskWeb`
建议 Monorepo：
- `UnionDeskWeb/apps/UnionDeskCustomerWeb`：客户端。
- `UnionDeskWeb/apps/UnionDeskAdminWeb`：管理端。
- `packages/shared`：共享 API Client、类型定义、公共组件与工具。

---

## 3. 逻辑架构

```text
[客户端 Web]        [管理端 Web]
      \               /
       \             /
        --- [反向代理 / Ingress] ---
                        |
                 [UnionDesk 后端]
 -------------------------------------------------
 | Auth | IAM | Domain | Ticket | Feedback | Notify |
 -------------------------------------------------
                        |
           [MySQL] [Redis] [对象存储] [消息队列(可选)]
```

说明：
- MVP 阶段可先使用反向代理，不强制独立 API Gateway。
- 通知发送建议异步化，避免阻塞主链路。

---

## 4. 后端模块设计（模块化单体）

1. `auth`：登录、令牌刷新、密码策略、验证码。
2. `iam`：用户、角色、域授权、权限校验。
3. `domain`：业务域配置、可见策略、状态管理。
4. `ticket`：工单类型、动态字段、状态流转、SLA。
5. `consultation`：咨询会话与转工单。
6. `feedback`：反馈/建议生命周期管理。
7. `notification`：站内信/邮件/短信模板与发送任务。
8. `audit`：审计日志、操作追踪、导出。

---

## 5. 前端架构建议（UnionDeskWeb）

- 技术栈基线：React + TypeScript + Ant Design（客户端与管理端统一组件体系）。

### 5.1 客户端
- 路由：登录、选域、提单、工单中心、反馈中心。
- 能力：动态表单、附件上传、进度追踪、消息提醒。

### 5.2 管理端
- 路由：工作台、工单池、工单详情、配置中心、权限中心。
- 能力：高级筛选、批量操作、SLA 看板、审计查询。

### 5.3 共享层
- 基于 OpenAPI 生成类型化 API Client。
- 统一错误处理与重试策略。
- 统一设计令牌与主题变量。

---

## 6. 接口与安全规范

### 6.1 API 规范
- 路径版本化：`/api/v1/...`
- 统一错误结构与稳定错误码。
- 创建/流转接口支持 `Idempotency-Key` 幂等控制。

### 6.2 鉴权与授权
- Access Token + Refresh Token。
- 后端计算有效业务域范围，不信任前端直接传权限范围。
- 对关键操作执行对象级权限校验。

### 6.3 数据与文件安全
- 附件通过签名 URL 控制访问。
- 上传时校验大小与类型，建议接入恶意文件扫描。
- 管理类接口统一审计。

---

## 7. 数据模型建议

### 7.1 身份、账号与角色
当前目标态采用 **统一身份主体 + 员工账号 + 客户账号**：
- `identity_subject`：统一身份主体，一个自然人只归属一个主体。
- `staff_account`：员工端、平台管理和业务域管理账号。
- `customer_account`：客户端、工单提报、咨询、知识访问和评价账号。
- `platform_role` / `staff_account_platform_role`：平台角色与员工账号授权；`platform_admin` 为平台级最高权限。
- `domain_role` / `domain_member_role`：业务域角色与域内成员授权；域级 `super_admin` 为单个业务域最高权限，低于平台级 `platform_admin`。

平台角色和业务域角色分属不同体系，域内数据权限通过业务域关系与权限项共同控制。

### 7.2 工单与动态字段
- 配置：`ticket_type`、`custom_field_config`。
- 运行态：`ticket`、`ticket_custom_field_value`（JSON）。
- 高频筛选字段（`status`、`priority`、`assigned_to`）保持独立列并建索引。

### 7.3 审计追踪
- `ticket_history`：工单状态、指派、回复、合并和系统动作。
- `audit_log`：配置、权限、管理和自助安全操作。
- `login_log`：登录成功/失败与来源信息。

## 8. 部署与环境规划

### 8.1 环境
- `dev`、`test`、`staging`、`prod`。

### 8.2 部署
- 后端容器化部署。
- 本期前端静态资源与附件优先通过源站 / 对象存储直链访问，预留 CDN 接入能力。
- 全链路 HTTPS。

### 8.3 配置与密钥
- 配置按环境隔离。
- 密钥统一托管（Vault/KMS 类服务）。
- 禁止本地文件存生产密钥。

---

## 9. 可观测性与运维

1. 指标：时延、错误率、SLA 达成率、队列积压、通知成功率。
2. 日志：优先接入 ELK（应用日志、业务日志、审计日志分层）。
3. 若环境暂未接入 ELK：使用 Spring 日志（Logback）输出本地滚动文件。
4. 告警：接口异常、慢查询、通知失败、队列堆积。
5. 备份：P0 提供备份脚本、恢复手册与一次恢复流程验证；定期恢复演练作为生产上线后的运维 Gate，明确 RPO/RTO。

---

## 10. 架构演进路线

- 阶段 1（MVP）：模块化单体 + 单库，快速验证与上线。
- 阶段 2（增长期）：按压力拆分通知、搜索、报表子系统。
- 阶段 3（规模化）：按领域拆分服务，建设事件总线与数据治理体系。
