# P0 开发方案

> 输入依据：`README.md`、`doc/产品需求文档 v1.0.md`、`doc/P0接口契约表.md`、`doc/P0验收用例表.md`、`doc/checklist/p0-execution/`、当前后端/前端工程结构与 Git 工作区状态。
> 目标：在不扩大 P0 范围的前提下，完成身份登录、业务域、RBAC、客户提单、客服处理、基础 SLA、基础通知、附件、健康检查与最小部署闭环。

## 1. 当前判断

| 维度 | 当前状态 | 研发判断 |
|---|---|---|
| 需求与验收 | PRD 已形成唯一真相源；P0 为 53 条必过用例，TC-019 为 P1 参考 | 可以进入 P0 研发拆解 |
| 后端基础 | Spring Boot + Flyway + MyBatis 工程已存在；认证、IAM、组织树已有实现与测试 | 先稳定启动链路和迁移链路，再扩业务 |

| 工程状态 | 工作区存在大量未提交变更与新增文件 | 开发前先做变更盘点和分批提交，避免互相覆盖 |

## 2. 研发原则

1. P0 冻结：只做 PRD §1.4 P0 和 `P0验收用例表.md` 的 53 条 P0 必过项。
2. 先稳定后扩展：后端启动、Flyway、登录、权限拦截不稳定时，不进入工单和 SLA 大开发。
3. 接口先行：所有前后端联调以 `P0接口契约表.md` 为准，接口字段变更必须同步 PRD、用例和 shared types。
4. 真实链路优先：demo 数据服务只可作为过渡，进入 Sprint 1 后必须替换为持久化领域服务。
5. 每个 Sprint 结束必须更新对应 checklist，并跑最小验证命令。

## 3. 阶段方案

| 任务 | 输出 | 验证 |
|---|---|---|
| 工作区变更盘点 | 将现有修改分为文档、后端 auth/IAM、前端平台页、杂项临时文件 | `git status --short` 分类清楚 |
| 后端启动链路修复 | MySQL + Redis + Flyway + Spring Boot 可稳定启动 | `docker compose up -d`；`.\mvnw.cmd spring-boot:run` |
| Flyway 迁移基线确认 | 迁移脚本顺序、幂等性、目标模型差异清楚 | `.\mvnw.cmd -q test`；readiness 反映 DB/Flyway |
| 认证/IAM 收口 | 登录、刷新、权限拦截、组织树接口与 AdminWeb 对齐 | auth/IAM 单测通过；AdminWeb 平台组织页可联调 |
| 外部依赖决策 | MinIO/SMTP 是否启用及降级模式冻结 | 决策写入 Sprint 0 checklist |

## 4. Gate 清单

| Gate | 通过标准 |
|---|---|
| G0 工作区 Gate | 当前未提交变更已分类，临时/孤儿文件有处理决定 |
| G1 启动 Gate | 后端、AdminWeb、依赖服务可在本地稳定启动，日志无阻塞错误 |
| G2 契约 Gate | `P0接口契约表.md` 中 P0 接口均有实现或明确任务归属 |
| G3 测试 Gate | 后端单测、前端 typecheck/lint、关键页面测试通过 |
| G4 验收 Gate | 53 条 P0 用例全部通过；TC-019 仅作为 P1 参考不阻塞 |

## 5. 近期优先级

1. 先完成 Sprint 0：启动/Flyway/auth/IAM/组织树稳定。
2. 再做 Sprint 1：身份、业务域、客户入域和平台管理员保活。
3. 然后集中 Sprint 2：工单闭环和附件链路。
4. 最后 Sprint 3：SLA、通知、安全与交付验收。
